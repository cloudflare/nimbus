`nimbus-docs add <slug>` installs from the [registry](/registry). It works in two modes depending on what you're adding — files copy in, or a recipe hands off to a coding agent.

## Components and utilities

For `registry:ui` and `registry:lib` slugs, the CLI resolves dependencies, copies files into your repo, and installs any npm packages they need:

<PackageManagers pkg="@cloudflare/nimbus-docs" type="dlx" args="add accordion" />

<Steps>
  <Step title="Resolve dependencies">
    The CLI walks the slug's `registryDependencies` and npm `dependencies` transitively.
  </Step>
  <Step title="Copy files">
    Each file lands at its declared path — `src/components/ui/<slug>/`, `src/lib/`, etc.
  </Step>
  <Step title="Register globals">
    Components used in MDX without an import must be added to `src/components.ts`. The CLI tells you which.
  </Step>
</Steps>

Once installed, the component lives in your repo — edit it freely. Pass `--yes` to skip overwrite prompts.

## Features

For `registry:feature` slugs there's nothing to copy. The recipe is a Markdown prompt a coding agent reads, adapts to your project, and applies:

```bash
nimbus-docs add 404-page
```

If a coding agent is detected in the environment, the CLI prints the recipe to stdout for it to consume. Otherwise pipe it yourself:

```bash
nimbus-docs add 404-page --print | claude
```

## Browse what's available

```bash
nimbus-docs list
nimbus-docs list --type ui
```

Every shipped component also has a showcase at [/components](/components) — preview, props, and the install command. See the [CLI](/cli) reference for all flags.
